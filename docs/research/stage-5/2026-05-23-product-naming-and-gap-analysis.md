## Research Phrase Bank

- [Verified] The strongest repeated user-language themes in the research corpus are:
  - `clean output you can actually use`
  - `trust`
  - `proof`
  - `what came back`
  - `what stayed readable`
  - `where the limits showed up`
  - `structure`
  - `fidelity`
  - `truth`
  - `signals`
- [Inference] The best product names for this market should sound like they belong to a trust-first extraction tool, not a general crawling platform or a browser-agent lab.

## Shipped Since This Ranking Started

- Cross-format structure recovery has now shipped and no longer belongs in the build-next queue. See `docs/superpowers/specs/2026-05-24-cross-format-structure-recovery-layer.md` for the completed lane definition.
- Homepage public proof has now shipped as a quiet inline counter directly below the primary action row on `/`.
- Authenticated extraction has been removed from the core homepage and `/batch` flows and moved into the standalone `pages-behind-login` route, so it no longer belongs in the main build-next queue.

## Build-Next Ranking

| Priority | Feature | User need it answers | Why it should come next | What it should entail |
| --- | --- | --- | --- | --- |
| 1 | Structured extraction mode | Users want trusted fields and machine-readable output after trust in the base extraction is established. | Competitors keep winning higher-value workflows with schema-shaped extraction. Oleriq already has trust and export groundwork; this is the next monetizable layer. | Schema templates, field previews, extraction confidence per field, provenance back to source spans or source blocks, and export to JSON/CSV alongside current reader outputs. |
| 2 | Hard-page evidence pack | Users want proof when a dynamic, partial, or broken result needs inspection. | Oleriq now tells users what state they are in, but it still lacks inspectable evidence on harder pages. | Screenshot capture, extraction-path summary, failure reason bundle, maybe DOM or text snapshots, and per-run evidence attached to the result state. |
| 3 | Deterministic batch reruns and presets | Teams want repeatable bulk jobs, clearer naming, and less manual re-setup. | `/batch` is now calmer and more trustworthy, so the next leverage point is repeatability. | Saved input presets, stable output naming rules, rerun-only-failed or rerun-only-partial actions, and better batch-history recall. |
| 4 | Protected-site escalation lane | Some users need coverage on harder blocked pages, but not at the cost of bloating the default product. | Demand is real, but so is complexity. This should stay an explicit escalation lane, not the default product promise. | Opt-in browser escalation, visible cost/risk messaging, proxy or remote-browser partnerships if needed, and a strict separation between normal extraction and protected-site attempts. |
| 5 | Narrow agent/API surface | Some users will want Oleriq inside existing agent workflows. | Demand exists, but it should follow stronger extraction truth and structured output rather than lead it. | Small API for extract/export/batch status first, then optional MCP or CLI packaging, with the same trust model and diagnostics exposed programmatically. |
| 6 | Search / crawl / map expansion | Some buyers will still want one product that can discover and collect across a site. | This is proven market demand, but it pulls Oleriq toward platform breadth. It should remain later than the trust-first gaps above. | URL discovery, crawl scopes, queue controls, search-plus-extract workflows, and site-level collection views only after the trust-first extraction lane is stronger. |
## Recommendation

- [Inference] The strongest naming direction is the `usable + proof` or `output + truth` family, because those are the cleanest translations of what users are actually asking for.
- [Inference] The strongest immediate product direction is still not "become the broadest crawler." It is "be the most trustworthy way to turn pages and supported documents into output people can actually use, then close the structured-extraction and evidence gaps."
